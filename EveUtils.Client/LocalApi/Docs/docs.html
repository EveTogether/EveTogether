Connect & Guides

Everything you need to build against the EVE Together local API. Base URL {{BASE_URL}}.

What this is

An opt-in, read-only HTTP API that runs inside the EVE Together client and exposes your own game data — live combat metrics, characters, fits, fleets and doctrines — so you can build stream overlays, widgets and tooling against it. It is one component; widgets are just one use case.

Security model

PropertyBehaviour
BindingListens on 127.0.0.1 only — never on a network interface. Not reachable from other machines.
AccessRead-only. There are no mutating endpoints; the API can never change your fits, fleets or settings.
SecretsNever exposes ESI tokens, granted scopes or credentials — only public game/fit/fleet data.
DefaultOff by default. Enable it and pick the port in Settings → Local API server.
HardeningHost-header validation rejects DNS-rebinding; CORS is open by default for this read-only loopback data, or restricted to a configured origin allowlist (localapi.allowedorigins); an optional API key (localapi.apikey) gates /api, /ws and /hub when set.

Getting started

Confirm the server is up, then explore the interactive reference:

# health check
curl {{BASE_URL}}/api/v1/health

# your characters (public identity only)
curl {{BASE_URL}}/api/v1/characters

From a browser widget (e.g. an OBS browser source):

const res = await fetch("{{BASE_URL}}/api/v1/metrics");
const metrics = await res.json();
// [{ characterId, characterName, dpsOut, dpsIn, neutPerSecond, capPerSecond, ... }]
Browser widgets run from a different origin — CORS is open for this read-only loopback data, so a fetch from any origin works. Optionally set localapi.apikey in Settings to gate access, then send it as ?key=… or the X-Api-Key header.

REST endpoints available

EndpointReturns
GET /api/v1/healthApp + API version.
GET /api/v1/metricsLive combat metrics for your running character(s), from the gamelog (always-on, fleet-independent).
GET /api/v1/charactersYour coupled characters: identity, corp/alliance, portrait URL, running state.
GET /api/v1/fitsYour fit library across both sources: your local fits and each coupled server's shared fits. Each row carries a scope ("local"/"server"), the server name and (for server fits) who shared it.
GET /api/v1/fits/{id}One fit's detail; add ?stats=true for the computed Dogma stats. For a server fit add ?server=<address> (its id is the server's shared-fit id).
GET /api/v1/fleetsYour fleets across both sources: client-only local fleets and, per coupled server, the active fleets you own or are in. Each row carries a scope ("local"/"server") and the server name.
GET /api/v1/fleetThe active fleet you are participating in: header, wing/squad structure and roster. 204 No Content when you are not in a fleet.
GET /api/v1/compositionsYour doctrine/composition library (local + each coupled server).
GET /api/v1/compositions/{id}One composition's roles + fit-entries. Add ?server=<address> for a server composition (omit for the local library).
GET /api/v1/types/{id}Resolve a type id to its name, group and a public icon URL — so a widget can show ship/module names + icons without its own SDE.
Live per-member fleet metrics are a streamed signal, not a readable snapshot — they arrive over the WebSocket stream (M3), not as a REST endpoint.

The full, interactive reference — request/response schemas and a live "try it" — is the Scalar API reference, generated from /openapi/v1.json.

Realtime: WebSocket available

A raw WebSocket at {{BASE_URL}}/ws — the recommended route for browser/OBS widgets, no client library needed. On connect you receive a snapshot, then a JSON event stream of { type, data, ts } (ts = unix ms):

{ "type": "snapshot",      "data": { "metrics": [ /* your toon(s) */ ], "fleet": { /* active fleet or null */ } }, "ts": ... }
{ "type": "metrics",       "data": [ /* your running toon(s), every second — always, fleet or not */ ],            "ts": ... }
{ "type": "fleet.metrics", "data": { "characterId":…, "fleetId":…, "kind":"Dps", "value":…, "unixMs":… },           "ts": ... }
{ "type": "fleet.changed", "data": { "fleetId":…, "kind":"…", "serverAddress":… },                                  "ts": ... }
const ws = new WebSocket("{{BASE_URL}}/ws".replace("http", "ws"));
ws.onmessage = (e) => { const evt = JSON.parse(e.data); /* evt.type, evt.data — update your overlay */ };
metrics streams your own running character(s) every second, fleet or not. fleet.metrics (one sample per member per tick) and fleet.changed only arrive while you are in a fleet. There is no fit.changed event yet.

Realtime: SignalR available

A SignalR hub at {{BASE_URL}}/hub/fleet for richer .NET/JS clients (@microsoft/signalr), with auto-reconnect and groups. Same events as the WebSocket stream, each as a named handler; you receive a snapshot on connect.

import * as signalR from "@microsoft/signalr";
const conn = new signalR.HubConnectionBuilder().withUrl("{{BASE_URL}}/hub/fleet").withAutomaticReconnect().build();
conn.on("snapshot",      (s)    => { /* { metrics:[…], fleet:… } on connect */ });
conn.on("metrics",       (rows) => { /* your toon(s), every second */ });
conn.on("fleet.metrics", (s)    => { /* one sample per member, in a fleet */ });
conn.on("fleet.changed", (c)    => { /* roster/state change, in a fleet */ });
await conn.start();
WebSocket (the section above) covers the browser/OBS case on its own with no dependency; SignalR is for richer .NET/JS clients that want auto-reconnect and a typed client library.
